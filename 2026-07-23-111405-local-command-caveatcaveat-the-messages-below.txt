  stripped in export builds.

⏺ This is a well-scoped feature, and the codebase already has all the seams it
  needs (the ABI append pattern, the with_world seam, the forward render pass
  with depth, the editor panel system). Before I scaffold, three decisions
  genuinely branch the implementation. Here's what I found and where I land on
  each.

  What's already settled (no decision needed):
  - ABI shape — this follows the existing append-only pattern exactly. Four new
  FerronApi entries: log_warn, log_error (mirroring the existing log), and
  debug_draw_line(from[3], to[3], color[4], duration). DrawRay is pure C#-side
  sugar over DrawLine (to = from + direction), so it needs no ABI slot. Append
  at the end of both FerronApi structs, never reorder — same rule as add_script.
  - Line lifetime — a DebugLines world resource holding Vec<{from, to, color, 
  expiry}>. Scripts push during their tick (through the deferred/leaf seam); the
  render pass reads it; entries with duration <= 0 live exactly one frame,
  others expire when time.elapsed >= expiry. Cleared/swept once per frame after
  extraction.

  The three real decisions:

  A. Log sink — "structured logging + visible in editor." Today ferron_log just
  does println!("[c#] …"). There's no log/tracing dependency and no console
  panel. So "structured + visible in editor" is net-new. My recommendation: a
  LogBuffer resource + a panels/console.rs, because it satisfies both halves of
  the requirement with machinery that matches how the rest of the editor already
  reads world resources.

  B. Where 3D debug lines render — the forward pass has a depth buffer
  (occlusion for free, reuses the camera view-proj) but is HDR so colors get
  tonemapped; a post-tonemap pass gives exact colors but has no depth attachment
  (no occlusion). This is the learning-critical renderer decision — I'll
  scaffold whichever you pick with signatures + TODOs and explain the pipeline,
  not write it wholesale.

  C. Strip mechanism — C#'s [Conditional] attribute strips call sites at compile
  time (truly zero-cost in export), vs. a runtime engine flag.

⏺ Short answer: just two fields — the subbuffer_allocator you added, plus the
  pipeline you're currently building but throwing away.

  pub struct LinePass {
      pipeline: Arc<GraphicsPipeline>,
      subbuffer_allocator: SubbufferAllocator,
  }

  Why those two and nothing else:

  - pipeline — bound in record. It also carries its own layout
  (pipeline.layout()), and since you derive the layout from_stages, the
  push-constant range from line.vert's Push { mat4 view_proj } is baked into it.
  So you push the VP through pipeline.layout() at draw time — no separate
  layout field.
  - subbuffer_allocator — this is your "reuse/grow a buffer across frames"
  mechanism. It sub-allocates a fresh Subbuffer<[LineVertex]> from a rotating
  pool each frame, which is exactly the per-frame streaming pattern (ForwardPass
  uses one the same way for its uniforms). So you don't need a standalone
  vertex-buffer field either — you mint one inside record.

  What you deliberately do not store:
  - No descriptor sets / set layouts — the line shader binds no resources, only
  a push constant.
  - No device / render_pass — only needed during construction.

  So your new should end with Self { pipeline, subbuffer_allocator }.

  Three things in the current draft will bite you the moment you wire it up, and
  they're tied to those fields:

  1. new returns () right now — you call GraphicsPipeline::new(...).unwrap(); as
  a statement and never build Self. Bind it to let pipeline = ...; and return
  the struct.
  2. vertex_input_state uses Vertex, not LineVertex. Vertex is the 5-attribute
  mesh vertex (position/normal/color/uv/tangent). Your buffer holds LineVertex
  (position + color), so it must be LineVertex::per_vertex().definition(&vs) or
  the vertex fetch is garbage.
  3. You're loading forward::vs/forward::fs. Those are the scene shaders —
  incompatible layout (they expect the full vertex, descriptor sets 0–3, and a
  big push-constant block). Give LinePass its own shader mods pointing at the
  files I dropped in:

  mod vs {
      vulkano_shaders::shader! { ty: "vertex", path: "shaders/line.vert" }
  }
  mod fs {
      vulkano_shaders::shader! { ty: "fragment", path: "shaders/line.frag" }
  }

  (That also spares you from making forward's private vs/fs mods pub.)

  vertex usage or record's bind_vertex_buffers will reject it —

❯ now sketch the record body                                                    

⏺ Here's the record body. It mirrors forward::draw's recording style (chained
  .unwrap()s, unsafe draw). The _builder/_camera/_extent params lose their
  underscores now that they're used.

  pub fn record(
      &mut self,
      builder: &mut AutoCommandBufferBuilder<PrimaryAutoCommandBuffer>,
      lines: &[DebugLine],
      camera: &Camera,
      extent: [u32; 2],
  ) {
      if lines.is_empty() {
          return;
      }

      // Two endpoints per segment (unchanged from the scaffold).
      let mut vertices: Vec<LineVertex> = Vec::with_capacity(lines.len() * 2);
      for line in lines {
          vertices.push(LineVertex { position: line.from.to_array(), color:
  line.color });
          vertices.push(LineVertex { position: line.to.to_array(),   color:
  line.color });
      }

      // Sub-allocate this frame's vertex buffer from the rotating pool and fill
  it.
      let buffer = self
          .subbuffer_allocator
          .allocate_slice::<LineVertex>(vertices.len() as u64)
          .unwrap();
      buffer.write().unwrap().copy_from_slice(&vertices);

      let aspect = extent[0] as f32 / extent[1] as f32;
      let view_proj = camera.view_projection(aspect).to_cols_array_2d();

      builder
          .set_viewport(
              0,
              [Viewport {
                  offset: [0.0, 0.0],
                  extent: [extent[0] as f32, extent[1] as f32],
                  depth_range: 0.0..=1.0,
              }]
              .into_iter()
              .collect(),
          )
          .unwrap()
          .bind_pipeline_graphics(self.pipeline.clone())
          .unwrap()
          .push_constants(self.pipeline.layout().clone(), 0, view_proj)
          .unwrap()
          .bind_vertex_buffers(0, buffer)
          .unwrap();

      let vertex_count = vertices.len() as u32;
      // SAFETY: the bound pipeline and vertex buffer cover [0, vertex_count); 
  no
      // index buffer or instancing is used.
      unsafe {
          builder.draw(vertex_count, 1, 0, 0).unwrap();
      }
  }

  A few things worth understanding rather than just pasting:

  - allocate_slice::<LineVertex>(len as u64) hands back a
  Subbuffer<[LineVertex]> from the allocator's pool. buffer.write() gives a
  host-write guard; copy_from_slice needs the lengths to match exactly, which
  they do since you allocated vertices.len(). This is why the allocator replaces
  a manual grow-a-Vec-buffer field — it recycles memory across frames for you.
  - Push constant is just [[f32;4];4]. view_proj (64 bytes at offset 0) lines up
  with line.vert's Push { mat4 view_proj }. No intermediate struct needed — a
  [[f32;4];4] implements BufferContents, so it goes straight through.
  self.pipeline.layout() is the layout that carries that push-constant range
  (derived from_stages), which is exactly why you didn't need a separate layout
  field.
  - set_viewport is required because the pipeline declares
  DynamicState::Viewport. Skip it and you'll get a validation error about unset
  dynamic state. forward::draw sets it every frame for the same reason.
  - draw is unsafe in this vulkano version (same as draw_indexed in forward).
  One LineList primitive is consumed per vertex pair, so vertex_count = 2 * 
  lines.len().

  You'll need one import: use vulkano::pipeline::graphics::viewport::Viewport;
  (you already import ViewportState, but not Viewport itself).
  bind_pipeline_graphics doesn't need PipelineBindPoint — that's only for
  descriptor sets, which you have none of.

  depth test is right, lines drawn inside a cube should be hidden by its front
  face.
